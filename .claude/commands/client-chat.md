# Client Chat Message

## Arguments
사용자 입력: $ARGUMENTS

## Context
- 개발팀의 영어 메시지를 클라이언트 카톡에 복붙할 한국어 메시지로 변환
- 질문인지 업데이트인지 자동 감지하여 적절한 포맷으로 출력
- 단순 번역이 아닌 재구성 — 기술 맥락을 클라이언트 관점으로 변환
- 노션 저장 없음 — 터미널에 바로 출력

## Mode Detection
입력 내용을 분석하여 자동으로 모드를 결정:

**Question mode** — 개발팀이 기획/정책/디자인 결정을 물어볼 때
- 키워드 신호: should, which, how, option, decide, confirm, clarify, need input, policy, or/or
- 또는 PM이 "확인", "질문", "물어봐" 등의 맥락을 제공할 때

**Update mode** — 개발팀이 진행 상황/완료/이슈를 보고할 때
- 키워드 신호: done, complete, deployed, in progress, blocked, fixed, merged, released, working on, ETA
- 또는 PM이 "업데이트", "보고", "진행상황" 등의 맥락을 제공할 때

**Mixed** — 둘 다 포함된 경우 → 업데이트 섹션 먼저, 질문 섹션 이어서

## Output Format

**기본: 짧은 메시지 (2-5문장)**
카톡 메시지는 짧을수록 좋다. 자연스러운 문장으로 작성.

구조:
1. 인사 + 용건 한 줄 (예: "안녕하세요, {project} 관련하여 공유드립니다.")
2. 핵심 내용 1-3문장
3. 질문이 있으면 질문 (1-2개면 문장으로, 3개 이상이면 번호)
4. 끝 — 질문이면 "확인 부탁드립니다." 업데이트면 별도 closing 없이 끝.

**구조화 포맷은 아래 경우에만:**
- 항목이 5개 이상
- PM이 명시적으로 "상세하게" 또는 "정리해서" 요청
- 구조화 시에도 볼드 헤더(`** 완료된 사항` 등) 대신 줄바꿈으로 구분

**금지:**
- 섹션 헤더 ("완료된 사항", "확인 요청 사항", "다음 단계" 등)
- 불필요한 번호 리스트
- 느낌표 ("공유드립니다!" → "공유드립니다.")
- "감사합니다!" 단독 closing
- [추론] 태그 — 클라이언트 메시지에 넣지 않음 (추론 필요 시 메시지 밖에서 PM에게 별도 언급)
- PM 확인 문구 ("수정할 부분 있으면 말씀해주세요") — 메시지 본문에 포함 금지

## Meeting-note Input Detection
미팅 노트 입력 시에도 짧은 메시지 원칙 동일. 미팅 내용을 전부 전달하지 않고, 클라이언트가 알아야 할 1-3가지만 추출.

입력에 meeting-note 구조 (미팅 정보/핵심 요약/이번 라운드 반영사항 등)가 포함되면 아래 추출 규칙 적용:

| meeting-note 섹션 | → client-chat 섹션 | 추출 규칙 |
|---|---|---|
| 핵심 요약 | 오늘 논의된 핵심 내용 | 고객이 알아야 하는 항목만. 내부 블로커/팀 리소스 제외. |
| 이번 라운드 반영사항 | 확정된 사항 + 진행 예정 | 고객 가시 항목만. 내부 버그 디테일은 결과로 압축 ("수정 후 안내드리겠습니다"). PM 항목 제외. |
| 후순위/유지보수 | (기본 제외) | 예외: 고객이 미팅에서 알고 있는 항목은 진행 예정에 한 줄로 ("~는 이후 단계에서 진행할 예정입니다"). |
| 고객/PM 확인 필요 | 추가 확인 부탁드리는 사항 | `Client:` 항목만. `PM:` 항목 제외. |

**제외 대상:** 🔴 기술 디테일, PM 내부 액션, 내부 기술 변경사항, 일정 지연 사유, 팀 리소스
**모드:** meeting-note 입력 시 Mixed (update + request)로 자동 판단

## Instructions

1. **인자 파싱**: 클라이언트명과 개발팀 메시지 내용 추출
   - `--client` 누락 시 `templates/client-default.md` 규칙 적용 (24h 내 activity-log의 client가 1개면 보수적 default 제안, 2+개면 숫자 선택지, 0개면 PM 확인)
2. **컨텍스트 로드** (아래 항목은 병렬 호출 가능 — 서로 독립):
   - `clients/{client-name}/CLAUDE.md` — 클라이언트 기술 수준, 커뮤니케이션 스타일, **언어** 파악
   - `glossary/{client-name}.md` — 용어 일관성 (영어→한국어 방향)
   - **언어 결정**: CLAUDE.md에 언어가 명시되어 있으면 (예: English) 해당 언어로 작성. 없으면 한국어 기본.
3. **모드 감지**: 입력 내용 분석하여 Question / Update / Mixed 판단
4. **메시지 작성**:
   - **Question mode**:
     - 배경: 개발팀이 왜 이걸 물어보는지 클라이언트가 이해할 수 있게 풀어서 설명
     - 확인 요청 사항: 기술 질문을 클라이언트가 답변할 수 있는 비즈니스/기획 질문으로 변환
     - 참고 사항: 선택지, 제안, 참고 정보가 있을 때만 포함
   - **Update mode**:
     - 완료/진행/이슈/예정으로 분류
     - 기술 용어 → 클라이언트가 이해할 수 있는 설명으로 변환
     - 시간 추정 → 캘린더 날짜로 변환 (가능한 경우)
     - 블로킹 항목 → 클라이언트가 답변할 수 있는 구체적 질문으로 변환
   - **Mixed mode**: 업데이트 → 질문 순서로 자연스럽게 연결
5. **톤**: 합니다체 기본. 클라이언트 CLAUDE.md에 별도 톤 지정이 있으면 그것을 우선 적용.
   - "~드립니다", "~있습니다", "~부탁드립니다"
   - 느낌표 사용 금지 — 마침표로 종결
   - 기술 용어는 반드시 쉬운 말로 풀어서 — 클라이언트 기술 수준에 맞춤
6. **한국어로 작성**: 카톡에 바로 복붙
7. **출력만**: 노션 저장하지 않음, 터미널에 바로 출력
8. **활동 로그**: 최종 메시지 출력 완료 시 기록 (취소/에러 시 미기록)
   ```bash
   echo '{"date":"'$(date +%Y-%m-%d)'","skill":"client-chat","client":"{클라이언트명}"}' >> .claude/activity-log.jsonl
   ```

## Rules
- 개발팀의 기술적 맥락을 클라이언트 관점으로 반드시 변환 (번역이 아니라 재구성)
- 질문이 여러 개면 번호 매겨서 정리 — 클라이언트가 하나씩 답변할 수 있게
- 개발팀이 선택지를 제시한 경우, 각 선택지의 장단점을 쉬운 말로 설명
- "이건 개발팀에서 알아서 하겠습니다" 류의 불필요한 정보는 필터링
- 업데이트에서 기술적 성취를 클라이언트 가치 관점으로 전환 (예: "API 리팩토링 완료" → "시스템 안정성 개선 완료")
- **Update mode 핵심**: 내부 개발 단위로 나열하지 말고, 고객이 체감하는 기능 단위로 그루핑하여 전달
  - (X) "키워드 관리, 배너 관리, 회원 관리, 블랙리스트 관리 완료" → 너무 specific
  - (O) "관리자 페이지 주요 기능 완료" → 고객 관점 요약
  - 프론트엔드/백엔드 분리하여 보고하지 않음 — 고객은 하나의 기능으로 인식
  - 전체 진행률(%)을 함께 안내 — 개발팀이 제공한 수치 기반으로 대략적 퍼센트 포함
  - 단, 블로커/확인 필요 사항은 구체적으로 유지
- 용어는 glossary 기준 사용
