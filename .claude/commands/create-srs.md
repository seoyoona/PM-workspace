# Create SRS (한국어 SRS/기능명세 초안 생성)

## Arguments
사용자 입력: $ARGUMENTS

## Context
- 스킬 목적: 개발 착수 전 정렬과 확인에 사용할 수 있는 수준의 한국어 SRS/기능명세 초안을 생성한다
- 입력은 세일즈 제안서, 미팅 메모, 고객 요청 문서, Notion 페이지 등 다양한 형태가 가능하다
- 학술적 SRS가 아닌, 실무에서 바로 개발 범위 정리/고객 확인/내부 문서화에 쓸 수 있는 문서가 목표

## 비창작 원칙 (No Invention Rule) — 최우선 안전장치

이 스킬의 모든 출력 단계에서 반드시 적용한다:

1. **입력 자료에 없는 기능은 임의로 생성하지 않는다.**
2. **일반적으로 있을 법한 기능이라도 소스에 명시되지 않았으면 본문에 넣지 않는다.**
   "보통 이런 서비스에는 XX가 있다"는 추론 사유로 불충분.
3. **소스의 직접 맥락상 필연적인 경우에만** `[추론]` 태그로 제한적 기술을 허용한다.
   - 허용 예: 소스에 "결제 기능" 명시 → `[추론] 결제 검증 로직 필요`
   - 금지 예: 소스에 "로그인" 명시 → "비밀번호 찾기"를 본문에 추가
4. **`[추론]`보다 `[확인 필요]`를 우선한다.** 판단이 애매하면 본문 대신 확인 필요로.
5. **소스에 없는 항목은 다음 중 하나로만 처리:**
   - 본문 제외 (아예 언급하지 않음)
   - `[확인 필요]` 섹션에 "미언급 / 포함 여부 미정"으로 표기

## Instructions

### Step 1: 입력 파싱 + 소스 수집
1. 사용자 입력에서 추출:
   - `--client`: 고객사명 (회사명). 필수 — 미입력 시 질문
   - `--project`: 프로젝트/서비스명. 선택 — 미입력 시 소스에서 추출 시도, 불명확하면 질문
   - 소스 자료: 텍스트 또는 Notion 링크
2. `glossary/{client-name}.md` 용어집 로드 (있으면)
3. `clients/{client-name}/CLAUDE.md` 클라이언트 컨텍스트 로드 (있으면)
4. Notion 링크가 있으면 MCP fetch로 내용 읽기
5. 접근 실패 시 사용자에게 알리고 중단

### Step 2: 요구사항 추출 + 분류

소스를 읽고 아래 순서로 처리:

1. **기능/요구사항 추출**: 소스에서 기능, 요구사항, 제약사항, 제외사항을 모두 식별
2. **서비스 구분**: User / Admin / System 등 자연스러운 구분이 있으면 분리
3. **기능 그룹핑**: 아래 기본 도메인 코드 셋 기준으로 분류

#### REQ ID 기본 도메인 코드 (고정)

| 코드 | 영역 |
|---|---|
| `API` | 일반 API (특정 도메인에 안 맞는 API) |
| `AUTH` | 인증, 회원, 권한 |
| `PAY` | 결제, 크레딧, 과금 |
| `DATA` | 데이터 저장, 조회, 내보내기 |
| `ADMIN` | 관리자 기능 |
| `UI` | 화면, UX (프론트 범위일 때) |
| `SYS` | 인프라, 시스템, 배포 |
| `AI` | AI/ML 관련 |
| `CONTENT` | 콘텐츠 관리, CMS |

- 기본 셋으로 먼저 분류한다
- 기본 셋으로 분류가 어색한 경우에만 커스텀 코드를 미리보기에서 PM 승인 후 사용한다
- REQ ID 형식: `REQ-{도메인}-{번호}` (번호는 01부터 순차)

4. **완성도 판단**: 각 항목에 대해:
   - 소스에 명시 → 그대로 기술
   - 소스 직접 맥락상 필연적 → `[추론]` 태그 (제한적)
   - 불명확/미언급 → `[확인 필요]` 섹션에만

### Step 3: 미리보기 (필수 — 바로 저장 금지)

**Notion에 바로 생성하지 않는다.** 반드시 아래 미리보기를 먼저 보여준다:

```
## SRS 초안 미리보기

프로젝트: {프로젝트명}
클라이언트: {고객사명}

### 섹션 구조
1. 프로젝트 개요
2. 서비스 배경 및 목표
3. 개발 범위
4. IA
5. 기능명세 요약
6. 상세 기능명세
7. 확인 필요 사항
(8. 용어 정의 — 해당 시)

### 기능 그룹
| 그룹 | 도메인 코드 | REQ 개수 |
|------|------------|---------|
| ... | ... | ...건 |
| **합계** | | **{n}건** |

- 확인 필요 항목: {n}건
- 제외사항: {n}건
- 커스텀 도메인 코드: {있으면 나열, 없으면 "없음"}
```

PM 응답 대기:
- "진행" → Step 4
- 수정 요청 → 반영 후 다시 미리보기
- "취소" → 중단

### Step 4: SRS 초안 생성

아래 구조로 한국어 SRS 초안을 작성한다:

#### 섹션 1: 프로젝트 개요
- 프로젝트명, 형태, 예상 기간 (소스에 있으면), 개발 목표
- 핵심 1줄 요약

#### 섹션 2: 서비스 배경 및 목표
- 왜 이 서비스가 필요한지 (2-3줄)
- 핵심 목표 (번호 리스트, 3개 이내)

#### 섹션 3: 개발 범위 (Scope of Work)
- **In-scope**: 이번 계약에서 하는 것
- **Out-of-scope**: 명시적으로 하지 않는 것
- **과업 원칙**: UI 동결, 추가 개발 간주 조건 등 (해당 시에만)

#### 섹션 4: IA (Information Architecture)
- 서비스 구조도 (텍스트 트리)
- User / Admin 등 구분이 있으면 분리
- 구조가 단순하면 생략 가능

#### 섹션 5: 기능명세 요약 (인덱스 역할만)
- 섹션별 REQ ID 테이블
- 컬럼: `REQ ID | 요구사항` (한 줄 이름만)
- **설명은 여기에 넣지 않는다** — 인덱스 역할만 수행

#### 섹션 6: 상세 기능명세
- 섹션별: 목적 → 구현 필요 기능 (bullet)
- 필요 시 Input/Process/Output 패턴
- 제외사항: 해당 기능 바로 아래에 인라인 `**제외:**` 라벨로 표기
- 섹션 5와 같은 설명이 반복되면 안 됨

#### 섹션 7: 확인 필요 사항 (Ambiguities / Open Questions)
아래 분류로 정리한다. PM이 고객에게 바로 물어볼 수 있는 질문 형태로 작성:

| 분류 | 설명 |
|---|---|
| **범위 확인** | 포함/제외 여부가 불명확한 기능 |
| **정책 확인** | 비즈니스 규칙/정책 미정 |
| **운영 확인** | 운영 방식/담당 미정 |
| **기술 전제** | 기술적 전제 조건 미확인 |

- 각 항목에 관련 REQ ID 참조 포함
- "확인 필요"라고만 적지 말고 "A인가 B인가?" 또는 "XX 포함 여부 미언급" 형태로

#### 섹션 8: 용어 정의 (조건부)
- 도메인 특화 용어가 있을 때만 출력
- 없으면 섹션 자체 생략
- 형식: `한국어 용어 | 영어 용어 | 비고`

### Step 5: 출력 (문서 길이 적응)

| REQ 총 개수 | 출력 방식 |
|---|---|
| ~15건 이하 | 전문 출력 |
| 16건 이상 | 핵심 발췌 (개요 + 개발 범위 + 기능명세 요약 + 확인 필요) + "전문이 필요하면 말해주세요" |

발췌 출력이라도 아래는 반드시 표시:
- 섹션 구조
- 기능 그룹 + REQ 총 개수
- 확인 필요 개수
- 제외사항 개수

### Step 6: Notion 저장 (PM 확인 후)

```
Notion 프로젝트 문서 DB에 저장할까요?
1. 저장 (추천)
2. 수정 후 저장
3. 저장 안 함
```

저장 시:
- DB: 프로젝트 문서 DB (`339823375b0c8182a853f7695648de84`) — Bash curl API
- 문서명(title): `SRS 초안`
- 클라이언트: `--client` 값
- 프로젝트: `--project` 값
- 유형: `SRS`
- 상태: `Draft`
- 언어: `한국어`
- 단계: `Sales Handover`
- 작성일: 오늘

```
다음 단계:

- 킥오프 준비 → /kickoff-prep
```

## Rules
- **비창작 원칙 최우선** — 소스에 없는 기능은 본문에 넣지 않는다
- **미리보기 필수** — 바로 Notion 저장 금지. PM 확인 후에만
- **중복 금지** — 기능명세 요약은 인덱스만. 상세에서만 설명
- **조건부 섹션** — 내용 없으면 섹션 자체 생략 (용어 정의, IA, 과업 원칙 등)
- **확인 필요 = actionable** — PM이 고객에게 바로 물어볼 수 있는 질문 형태
- **제외사항 인라인** — 관련 기능 바로 아래에 `**제외:**`로 표기. 별도 섹션 분리 안 함
- **용어 일관성** — glossary에 정의된 용어만 사용
- **REQ 도메인 고정** — 기본 9개 코드 우선. 커스텀은 미리보기에서 PM 승인 후만
- **한국어 출력** — SRS 본문은 한국어. 기술 용어는 영어 병기 허용 (예: "크레딧 차감 (Credit Deduction)")
- **톤** — 합니다체(격식체). 명료하고 간결하게. 불필요한 수식 없이
